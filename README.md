# 국내 주식 스윙(눌림목) 추천 시스템

상승 추세인 종목이 **20일 이동평균선까지 눌렸다가 반등하는 첫 자리**(눌림목)를
자동으로 스캔하고, 향후 **텔레그램 알림 + 웹 대시보드**로 실시간 추천하기 위한
프로젝트입니다.

> ⚠️ **이 시스템은 종목 "추천"만 제공하며 실제 매매는 본인의 판단과 책임으로
> 별도 계좌에서 진행해야 합니다.** 어떤 자동 매수/매도 주문 기능도, 증권사 계좌
> 접근 코드(KIS API 등)도 포함하지 않습니다. 표시되는 점수·목표가·손절가는 전략
> 규칙에 따른 참고용 예시일 뿐 매매 권유가 아니며, 실제 투자 전 반드시 백테스트로
> 검증하고 본인 책임하에 판단하세요.

## 현재 단계: 전략 + 백테스트 ✅

| 구성 | 파일 |
|------|------|
| 데이터 수집 (실데이터/합성) | `swing/data.py` |
| 눌림목 전략·신호 생성 | `swing/strategy.py` |
| 백테스트 엔진 (매매 추출·포트폴리오) | `swing/engine.py` |
| 성과 지표·리포트 | `swing/metrics.py` |
| 실행기 | `run_backtest.py` |
| 무결성 테스트 | `tests/test_engine.py` |

### 눌림목 매수 규칙 (요약)

**진입 조건** (종가 확정 후 판단 → 다음 봉 시가 체결, 미래참조 없음)
1. 추세: 종가 > 60일선, 20일선 > 60일선(정배열), 60일선 우상향
2. 눌림: 당일 저가가 20일선 +2% 이내로 근접(단, 60일선은 안 깨짐), 직전 20일 고점이 현재가보다 8%+ 높음
3. 반등: 종가가 20일선 위로 회복 + 양봉 + RSI 40~68 + 거래량 ≥ 20일 평균×0.8

**청산 우선순위**: 손절 −5% → 익절 +10% → 20일선 −3% 이탈 → 15거래일 초과
왕복 비용(수수료+세금+슬리피지) 약 0.35% 반영.

모든 수치는 `swing/strategy.py`의 `PullbackParams`에서 조정할 수 있습니다.

## 실행 방법

```bash
pip install -r requirements.txt

# 1) 실제 국내 데이터로 백테스트  (네트워크 열린 환경: 내 PC / GitHub Actions)
python run_backtest.py --source real --market KOSPI --top 100 --start 2020-01-01

# 2) 오프라인 검증  (합성 데이터, 네트워크 불필요)
python run_backtest.py --source synthetic --n 40 --days 900

# 테스트
python tests/test_engine.py
```

출력물:
- 콘솔 성과 리포트 (승률·손익비·기대값·CAGR·MDD 등)
- `trades.csv` — 개별 매매 내역
- `signals_latest.json` — 가장 최근 봉에서 신호가 뜬 **오늘의 후보** (알림/웹 연동용)

### ⚠️ 네트워크 관련 참고
이 저장소를 만든 원격(웹) 세션에서는 보안 정책상 네이버·KRX·야후 등 증시 데이터
호스트 접근이 차단되어 **실데이터 백테스트는 여기서 돌릴 수 없습니다.** 그래서
합성 데이터로 엔진을 검증했습니다. `--source real`은 **네트워크가 열린 회원님 PC나
GitHub Actions에서** 정상 동작합니다.

## 2단계: 텔레그램 알림 + 자동화 ✅

| 구성 | 파일 |
|------|------|
| 텔레그램 전송 (표준 라이브러리) | `swing/notify.py` |
| 장중 스캔 + 신규 알림 (중복 방지) | `run_scan.py` |
| 무료 스케줄러 | `.github/workflows/swing-scan.yml` |
| 상태(중복방지·최신 신호) | `state/alerted.json`, `state/signals_latest.json` |

### 텔레그램 봇 설정 (5분)

1. 텔레그램에서 **@BotFather** → `/newbot` → **봇 토큰**(`123456:ABC...`) 발급
2. 만든 봇과 대화 시작 후 **@userinfobot** 으로 본인 **chat_id**(숫자) 확인
3. 로컬 테스트:
   ```bash
   export TELEGRAM_TOKEN="봇토큰"
   export TELEGRAM_CHAT_ID="내chat_id"
   python run_scan.py --market KOSPI --top 100          # 실데이터 + 전송
   python run_scan.py --source synthetic --dry-run       # 미전송 테스트
   ```

### GitHub Actions 자동 실행 (무료)

1. 저장소 **Settings → Secrets and variables → Actions** 에 두 개 등록:
   `TELEGRAM_TOKEN`, `TELEGRAM_CHAT_ID`
2. 워크플로우가 **평일 장중(KST 09:00~15:30) 30분마다** 자동 스캔 → 신규 신호를 텔레그램 전송
3. 스캔 결과는 `state/` 에 커밋되어 중복 알림을 방지하고, 이후 웹 대시보드가 이를 읽습니다.

> ⚠️ **예약(schedule) 실행은 기본 브랜치(`main`)에 워크플로우가 있어야만 동작**합니다.
> 지금은 작업 브랜치에 있으므로, `main` 에 병합하면 자동 실행이 시작됩니다.
> 병합 전에는 Actions 탭의 **Run workflow(workflow_dispatch)** 로 수동 실행할 수 있습니다.

`dashboard.html` — 휴대폰/PC 어디서든 보기 좋게 만든 반응형 웹 대시보드입니다.
**5단계에서 일일 스코어링 순위표 + 날짜 드롭다운** 형태로 업그레이드되었습니다
(아래 5단계 참고). `results/` 의 CSV 히스토리를 읽어 최신 스캔과 과거 스캔을
모두 볼 수 있습니다.

> 장중 실시간 신호(`run_scan.py`)는 여전히 텔레그램으로 전송되며
> `state/signals_latest.json` 에 최신 스냅샷이 남습니다. 웹 순위표는 장마감 후
> 하루 1회 집계되는 일일 스코어(5단계)를 기준으로 합니다.

### GitHub Pages 로 공개하기 (무료)

1. 저장소 **Settings → Pages** → Source: `Deploy from a branch` → Branch: `main` / `/ (root)` 저장
2. 잠시 후 `https://<사용자명>.github.io/etf-partner-finder/dashboard.html` 로 접속
3. 일일 스캔(5단계)이 `results/` 를 갱신할 때마다 대시보드에 자동 반영됩니다

> 로컬에서 미리 보려면: `python -m http.server 8000` 실행 후
> `http://localhost:8000/dashboard.html` 접속 (파일 직접 열기는 fetch 제약으로 동작 안 함).

## 4단계: 전략 튜닝 (과최적화 방지) ✅

`tune.py` — 파라미터 조합을 격자 탐색하되, 데이터를 **앞(학습)/뒤(검증)** 로 나눠
**검증 구간에서도 성적이 유지되는 강건한 조합**만 상위로 올립니다. 학습에서만 좋고
검증에서 무너지는 조합은 과최적화이므로 걸러집니다.

```bash
# 실데이터 (네트워크 열린 환경) — 기간을 길게 줄수록 신뢰도 ↑
python tune.py --source real --market KOSPI --top 100 --start 2019-01-01

# 오프라인 검증
python tune.py --source synthetic --n 60 --days 1000
```

- 탐색 격자는 `tune.py`의 `GRID` 에서 조정 (조합 수가 곱으로 늘어나니 과하게 넓히지 말 것)
- 표본이 `MIN_TRADES`(기본 20) 미만인 조합은 신뢰 불가로 제외
- **고르는 법**: 검증(test) 기대값이 (+)이고 학습·검증 성적이 크게 어긋나지 않는 조합

### 튜닝 레버 (전략에 추가된 선택 옵션)

`PullbackParams` 에 ATR 기반 청산과 트레일링 스톱을 넣어 탐색 폭을 넓힐 수 있습니다
(기본은 비활성 = 기존 고정 % 청산):

| 파라미터 | 의미 |
|----------|------|
| `use_atr_exits` | 손절/익절을 진입시점 ATR 배수로 산정 |
| `atr_stop_mult` / `atr_target_mult` | ATR 손절/익절 배수 |
| `trail_atr_mult` | >0 이면 이익 구간에서 최고종가 대비 ATR 배수 이탈 시 청산 |

## 5단계: 매일 자동 종목 스코어링 + 히스토리 대시보드 ✅

이진(매수/관심) 신호를 넘어, **상승추세 종목을 0~100점으로 줄세워** 매일 순위표를
남깁니다. 장마감 후 하루 1회 자동 실행되어 결과를 CSV 히스토리로 커밋합니다.

> 🔄 **6단계에서 스코어링 엔진이 눌림목 → 모멘텀 돌파로 교체되었습니다.** 아래
> 스코어링 로직 설명(`ScoreParams`)은 초기 눌림목 버전 기준이며, 현재
> `run_daily_scan.py`·`dashboard.html`은 6단계의 모멘텀 전략(`swing/momentum.py`)을
> 사용합니다. 자동화·CSV 히스토리·대시보드 구조는 그대로입니다.

| 구성 | 파일 |
|------|------|
| 후보 스코어링(총점 + 하위점수) | `swing/score.py` |
| 일일 스캔 실행기 (CSV 저장 + 상위5 알림) | `run_daily_scan.py` |
| 매일 자동 실행 (평일 15:35 KST) | `.github/workflows/daily-scan.yml` |
| 결과 히스토리 | `results/scan_YYYYMMDD.csv`, `results/index.json` |
| 순위표 대시보드 (날짜 선택) | `dashboard.html` |

### 총점은 어떻게 매기나 (`ScoreParams`)

상승추세(종가>60일선·정배열·60일선 우상향)인 종목만 대상으로, 네 개 하위 점수를
가중 평균해 **총점(0~100)** 을 냅니다.

| 하위 점수 | 의미 | 기본 가중치 |
|-----------|------|:-----------:|
| **추세(trend)** | 60일선 위 여유 + 정배열 간격 + 60일선 기울기 | 0.35 |
| **눌림(pullback)** | 종가가 20일선에 근접할수록 높음(눌림 자리) | 0.30 |
| **모멘텀(momentum)** | RSI 가 반등 여력 구간(과열·과매도 아님)인지 | 0.20 |
| **거래량(volume)** | 최근 거래량이 20일 평균 대비 실린 정도 | 0.15 |

가중치·기준값은 `swing/score.py` 의 `ScoreParams` 에서 조정합니다.

> 병합 전에는 Actions 탭의 **Run workflow(workflow_dispatch)** 로 수동 실행하세요.

GitHub Pages 배포 방법은 3단계와 동일합니다

## 6단계: 모멘텀 돌파 전략 + 백테스트 최적화 ✅

**"며칠~수주간 강한 상승을 노리고, 손실은 짧게·수익은 길게"** 목표에 맞춘 새 전략입니다.
눌림목(평균회귀)과 방향이 반대라 별도 모듈로 새로 짰습니다.

| 구성 | 파일 |
|------|------|
| 모멘텀 돌파 전략 + 랭킹 점수 | `swing/momentum.py` |
| 백테스트 최적화 (학습/검증 분리) | `optimize_momentum.py` |
| 실데이터 최적화 자동 실행 | `.github/workflows/optimize.yml` |
| 최적화 결과 (일일 스캔이 자동 사용) | `results/momentum_best.json` |

### 전략 규칙

- **진입**: `entry_lookback`일 **신고가를 종가로 돌파** + 장기추세선(`ma_trend`) 위 +
  추세선 우상향 + 모멘텀(ROC)≥0 + 거래량 ≥ 20일평균×`vol_mult` + RSI 밴드 안
  → **다음 봉 시가 체결**(미래참조 없음)
- **손실 짧게**: 진입 즉시 **초기 손절 = 진입가 − `init_stop_atr`×ATR** (작게)
- **수익 길게**: 고정 익절 없음. **샹들리에 트레일 = 보유중 최고가 − `trail_atr`×ATR**
  로 추세를 끝까지 탄다. 종가가 `exit_lookback`일 최저가 이탈 시에만 추세붕괴 청산

### 랭킹 점수 (대시보드 총점)

장기추세 위 종목을 **추세·돌파·모멘텀·거래량·RSI** 5개 하위 점수(각 0~100)의 가중
평균으로 줄세웁니다. **가중치(비율)와 각 로직의 기준값까지 백테스트로 최적화**합니다.

### 백테스트 최적화 (2단계, 과최적화 방지)

```bash
# 실데이터 (네트워크 열린 환경) — 학습/검증 분리로 강건한 조합만 채택
# 기본 목적함수 = return(총수익). 포트폴리오 누적수익률을 가장 크게 하는 조합을 고른다.
python optimize_momentum.py --source real --market KOSPI --top 150 --start 2018-01-01 \
    --objective return --regime --max-combos 400

# 오프라인 코드 검증 (합성 데이터 — 수치는 참고 불가)
python optimize_momentum.py --source synthetic --n 80 --days 1200 --max-combos 200
```

- **1단계**: 진입 돌파창·추세선·거래량·RSI·ATR 손절/트레일·보유일 등 **전략 파라미터**를
  격자 탐색. 각 종목 시계열을 앞(학습)/뒤(검증)로 나눠 **검증에서도 유지되는** 조합만 채택.
  강건 가드로 **학습구간에서도 수익(obj>0)인 조합만** 신뢰한다(과최적화 배제)
- **2단계**: 그 전략으로 **랭킹 점수 가중치**를 탐색해, '점수가 높을수록 실제 수익이
  좋았는지'를 순위상관(Spearman)으로 학습·검증에서 확인
- `--objective` 로 무엇을 최대화할지 선택:
  - **`return`(기본)**: 포트폴리오 **누적수익률(총수익)**. `--max-positions`(기본 5)로
    동시 보유 종목 수를 정해 자본에 태워 시뮬한 뒤 검증구간 총수익이 최대인 조합을 고른다
  - `expectancy`(거래당 기대값) / `winrate`(승률) / `blend`(승률 하한 + 기대값)
- **생존편향 완화**: `--include-delisted` 를 주면 현재 상장 종목뿐 아니라 그동안
  **상장폐지된 종목(패자)까지 유니버스에 되살려** 백테스트한다. 현재 상장 종목만 쓰면
  성과가 부풀려지므로, 실데이터 최적화(`optimize.yml`)는 이 옵션을 기본으로 켠다.
  (폐지 목록을 못 가져오면 경고를 남기고 상장 종목만으로 진행)
- 결과는 `results/momentum_best.json` 에 저장되고, **일일 스캔이 다음 실행부터 이
  파라미터·가중치를 자동으로 사용**합니다
- `.github/workflows/optimize.yml` 의 **Run workflow** 로 실데이터 최적화를 돌릴 수 있습니다
  (무겁기 때문에 수동 + 월 1회 자동). 목적함수·승률하한·기간을 입력으로 조정

### ⚠️ "승률 70%"에 대한 솔직한 안내 (꼭 읽어주세요)

이 전략은 **추세추종**이라 구조적으로 **승률이 낮고(보통 35~50%) 손익비가 높습니다.**
"수익을 길게 가져간다(트레일링)"는 곧 **자잘한 실패 돌파에서 여러 번 −작게 잘리고,
가끔 크게 먹는 소수의 큰 수익**으로 돈을 버는 방식이기 때문입니다. 즉 **"승률 70% +
수익 길게"는 서로 모순**입니다. 승률을 억지로 70%로 끌어올리려면 익절을 짧게 잘라야
하는데, 그러면 "수익 길게"가 깨지고 큰 상승을 놓쳐 총 기대값이 낮아집니다.

- 승률을 우선하고 싶으면 `--objective winrate` 로 최적화해 **실제 트레이드오프를 직접
  확인**하세요. 다만 그 결과가 총수익까지 좋다는 보장은 없습니다.
- **추세추종에서 정말 중요한 건 승률이 아니라 손익비×기대값**입니다. 승률 40%라도
  손익비 2.5면 장기적으로 크게 이깁니다.
- 어떤 백테스트도 **미래 수익을 보장하지 않습니다.** 반드시 검증 구간 성적을 확인하고,
  소액으로 먼저 검증한 뒤, 본인 판단·책임으로 별도 계좌에서 진행하세요.

> 📌 **이 저장소를 만든 원격 세션은 증시 데이터 호스트가 차단돼 실데이터 최적화를
> 여기서 돌릴 수 없습니다.** 코드는 합성 데이터로 동작을 검증했고, **실제 최적화
> 수치는 위 `optimize.yml` 워크플로우(또는 네트워크가 열린 내 PC)에서 생성**됩니다.

## 7단계: 코스피 단기 전망 대시보드 ✅

개별 종목 스코어링(3~6단계)과 별개로, **지수(코스피) 자체의 단기 방향**을 체계적으로
전망하는 대시보드입니다. 다음 6팩터 분석 프레임을 그대로 코드화했습니다.

| 순서 | 팩터 | 무엇을 보나 (비중) |
|------|------|-----------|
| 1 | 🌎 거시 | 미 S&P500 추세·VIX·미10년물 금리·달러/원 (20%) |
| 2 | 🇰🇷 한국경제 | 원화 강도·중국 증시·코스닥 상대강도·**반도체 상대강도** (15%) |
| 3 | 🏭 기업실적 | **삼성전자·SK하이닉스 실적사이클 프록시** + 코스피 PER 방향 (15%) |
| 4 | 💰 수급 | 외국인·기관 순매수(대금) 방향과 강도 (20%) |
| 5 | ⚖️ 밸류에이션 | PER·PBR 역사적 백분위 + ERP(주식위험프리미엄) (10%) |
| 6 | 📈 기술적 | 정배열·RSI·MACD·모멘텀·거래량·52주 위치 (20%) |

각 팩터를 **0~100점(50=중립)** 으로 환산하고, 가중 평균으로 **종합 전망 점수**를 낸 뒤
방향(강세/중립-강세/중립/중립-약세/약세)과 신뢰도로 매핑합니다. 단기 전망이라 기술·수급
비중을 높였습니다. **코스피는 반도체 비중이 절대적**이므로 삼성전자·SK하이닉스 상대강도를
한국경제·실적 팩터의 핵심 변수로 함께 봅니다.

| 구성 | 파일 |
|------|------|
| 6팩터 전망 엔진 | `swing/kospi_forecast.py` |
| 전망 산출 + JSON/CSV 저장 러너 | `run_kospi_forecast.py` |
| 전망 대시보드 (반응형) | `kospi.html` |
| 매일 자동 갱신 | `.github/workflows/kospi-forecast.yml` |
| 최신 전망 + 점수 히스토리 | `results/kospi_forecast.json` · `results/kospi_history.csv` |

### 실행 방법

```bash
# 실데이터 (네트워크 열린 환경) — FDR(지수·환율·해외지수) + pykrx(PER/PBR·수급)
python run_kospi_forecast.py --start 2022-01-01

# 첫 실행 시 최근 120일 점수를 소급 계산해 히스토리(스파크라인)를 채움
python run_kospi_forecast.py --start 2022-01-01 --backfill 120

# 오프라인 검증 (합성 데이터, 미전송) — 파이프라인/대시보드 동작 확인용
python run_kospi_forecast.py --source synthetic --backfill 120 --dry-run
```

- 데이터 소스가 막히거나 일부 조회가 실패해도, 해당 신호를 **중립(50)·미가용**으로 두고
  파이프라인이 죽지 않습니다(대시보드에 흐리게 표시).
- 소급 점수는 **각 날짜 시점의 데이터만으로(미래 참조 없이)** 재계산합니다.
- `kospi.html` 은 `results/kospi_forecast.json`·`kospi_history.csv` 를 읽어 게이지·6팩터
  카드·점수/코스피 추이 스파크라인을 그립니다. GitHub Pages 배포는 3단계와 동일
  (`https://<사용자명>.github.io/etf-partner-finder/kospi.html`).

> ⚠️ **참고용 전망 점수**일 뿐 매매 권유가 아닙니다. 전망은 확률적 판단으로 언제든
> 빗나갈 수 있고, 자동 매수/매도 기능은 없습니다. 이 저장소를 만든 원격 세션은 증시
> 데이터 호스트가 차단돼 합성 데이터로 동작만 검증했으며, **실제 전망 수치는 위
> 워크플로우(또는 네트워크가 열린 내 PC)** 에서 생성됩니다.

### 업데이트 주기 — 실데이터로 매일 자동 갱신하는 법

전망 모델은 **일봉 + 장마감 후 확정되는 지표(PER/PBR·투자자 수급)** 로 계산하므로,
**하루 1회 장마감 후 갱신**이 자연스러운 주기입니다(진짜 실시간 틱은 유료 데이터가
필요하고, 일봉 기반 전망에는 큰 의미가 없습니다). 켜는 순서:

1. **이 브랜치를 `main` 에 머지** — GitHub 예약(schedule)은 기본 브랜치에 워크플로우
   파일이 있어야만 동작합니다.
2. 저장소 **Settings → Actions** 에서 Actions 를 활성화(필요 시).
3. (선택) **Settings → Secrets** 에 `TELEGRAM_TOKEN`·`TELEGRAM_CHAT_ID` 를 넣으면
   매일 전망 요약이 텔레그램으로도 옵니다.
4. (선택, **수급·밸류에이션 팩터 활성화**) **Settings → Secrets** 에 `KRX_ID`·`KRX_PW`
   (무료 [KRX 데이터포털](http://data.krx.co.kr) 계정)를 넣으세요. 현재 `pykrx` 는 KRX
   포털 **로그인(계정)** 이 있어야 지수 **PER/PBR** 과 **외국인·기관 투자자 수급**을
   내려받습니다. 계정이 없으면 이 두 팩터는 자동으로 **미가용(중립)** 처리되고 나머지
   (거시·한국경제·실적·기술) 4개 팩터는 실데이터로 정상 동작합니다.
5. **Actions → 코스피 단기 전망 → Run workflow** 를 `backfill=120` 으로 한 번 실행 →
   실데이터로 최신 전망 + 최근 120일 점수 히스토리가 채워집니다.
6. 이후 **평일 장마감(KST 15:40, UTC 06:40)마다 자동 실행**되어 `results/` 를 커밋하고
   `kospi.html` 대시보드에 바로 반영됩니다.

> 더 자주 보고 싶으면 `kospi-forecast.yml` 의 `cron` 에 장중 시간대를 한 줄 더 추가하면
> 되지만(예: 점심 무렵), 일봉 확정 전 값은 잠정치입니다.

## 8단계: 전망 가중치 최적화 (단기 예측력 캘리브레이션) ✅

**"저 점수 비율(20/15/15/20/10/20)은 최적화된 값인가?"** — 기본값은 *아닙니다*. 단기일수록
기술·수급이 더 잘 듣는다는 도메인 논리로 정한 **합리적 초기값(prior)** 입니다. 이를
**데이터로 캘리브레이션**해, 종합점수가 **단기(기본 60거래일≈3개월) 코스피 미래수익률을
가장 잘 예측**하도록 가중치를 맞추는 최적화기를 추가했습니다.

| 구성 | 파일 |
|------|------|
| 가중치 최적화기 | `optimize_kospi.py` |
| 최적화 결과 (전망이 자동 사용) | `results/kospi_weights_best.json` |
| 월 1회 자동 최적화 | `.github/workflows/optimize-kospi.yml` |

### 원리

- **목적함수 = 순위 IC(Information Coefficient)**: 각 시점의 종합점수와 그 뒤 `h`거래일
  수익률의 **스피어만 순위상관**. "점수가 높을수록 실제로 더 올랐는가"를 잰다.
- **미래참조 없음**: 각 날짜 `t` 의 팩터점수는 **그 시점 데이터만으로** 계산(`slice_bundle`),
  라벨은 `t→t+h` 미래수익. 예측력을 정직하게 측정한다.
- **워크포워드 검증**: 시계열을 앞(학습 65%)/뒤(검증 35%)로 나눠, **학습 IC 상위 후보 중
  검증(=미래) IC 가 가장 높은** 조합을 고른다 → 과최적화 완화.
- **퇴화 방지**: 한 팩터 몰빵(코너 해)을 막는 **가중치 상한(`--max-weight`)** + 선택값을
  기본 prior 로 당기는 **축소추정(`--shrink`)**. 다요소 성격과 강건성을 유지한다.
- `--horizon` 으로 목표 예측기간(거래일)을 정한다. **단기 ~3개월이면 60 근처**(20·40·60을
  함께 리포트). 결과는 `results/kospi_weights_best.json` 에 저장되고, **다음 전망 실행부터
  자동 적용**된다(`run_kospi_forecast.py` 의 `load_weights`).

### 실행 방법

```bash
# 실데이터 (네트워크 열린 환경) — 기간이 길수록 표본↑·신뢰도↑
python optimize_kospi.py --source real --start 2016-01-01 --horizon 60

# 오프라인 코드 검증 (합성 데이터 — 수치는 참고 불가, 동작만 확인)
python optimize_kospi.py --source synthetic --days 900 --samples 600 --step 2
```

> ⚠️ **단기 지수 예측은 노이즈가 매우 큽니다.** IC 는 보통 작고(0.05~0.15면 유의미),
> 어떤 백테스트도 미래 수익을 보장하지 않습니다. 합성 데이터에서는 **검증 IC 가 음수로
> 나오는데, 이는 "신호 없는 데이터엔 예측력이 없다"는 정직한 결과**입니다(워크포워드
> 검증이 노이즈를 억지로 맞추지 않음). 의미 있는 가중치는 **실데이터 최적화**에서 나옵니다.

## 9단계: 전망 점수 백테스트 (워크포워드 검증) ✅

**"이 전망 점수 체계(기술+수급+PER/PBR+…)로 매수/매도 시그널을 줬다면 과거에 실제로
수익이 났을까? 단순보유보다 나았을까?"** 를 검증한다.

| 구성 | 파일 |
|------|------|
| 백테스트 엔진 | `backtest_forecast.py` |
| 결과 (대시보드가 표시) | `results/kospi_backtest.json` |
| 월 1회 자동 백테스트 | `.github/workflows/backtest-kospi.yml` |

### 방식 (정직성)

- **미래참조 없음**: 각 날짜 `t` 의 6팩터 점수는 `t` 까지 데이터로만 계산(`score_panel`).
  시그널은 `t` 종가로 판단하고 **다음 거래일에 반영**(지연 체결).
- **시그널(롱/현금)**: 종합점수 ≥ `buy_thr` 매수, ≤ `sell_thr` 현금(매도), 밴드 안은 유지
  (히스테리시스). 포지션이 바뀌는 날 **왕복 비용**을 차감.
- **인샘플 튜닝 배제**: 가중치는 **기본값(prior)** 을 쓴다. (같은 기간에 튜닝한 가중치를
  쓰면 미래참조가 되므로 제외)
- **리포트**: 전략 vs **단순보유(buy-and-hold)** 총수익·CAGR·**초과수익**, **MDD·승률·
  샤프비율**·시장노출·거래횟수. 대시보드에 자산곡선(전략 vs 보유)과 지표 타일로 표시.

```bash
python backtest_forecast.py --source real --start 2016-01-01   # 실데이터(6팩터)
python backtest_forecast.py --source synthetic --days 1200      # 오프라인 코드검증
```

> ⚠️ **과거 성과는 미래를 보장하지 않습니다.** 단기 지수 타이밍은 노이즈가 크고,
> **표본기간·임계값에 민감**합니다. 예컨대 강한 상승장 구간만 보면 "현금 회피"가 있는
> 어떤 타이밍 전략도 단순보유를 이기기 어렵습니다. 이 백테스트는 시스템의 성격을
> 이해하기 위한 참고 도구이지 수익 보장이 아닙니다.
